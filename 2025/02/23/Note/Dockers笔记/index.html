目录挂载

**目录挂载：允许将宿主机上的一个文件或目录直接挂载到容器内。**这种方式灵活且配置简单，适合开发环境中的快速迭代，因为它能让开发者直接在宿主机上编辑代码，并立即在容器内看到效果

优点：

  • 灵活性高：可选择任意宿主机路径进行挂载。
  • 实时同步：修改宿主机文件可即时反映到容器中。

缺点：

  • 依赖性强：与宿主机文件系统紧密耦合，迁移性较差。
  • 性能影响：在某些操作系统（如Windows的NTFS或Mac的HFS+）上可能降低性能

实例：

数据卷

数据卷：Docker管理的独立存储空间，默认位于/var/lib/docker/volumes/下，但对用户透明。它更适合生产环境，提供更好的安全性和性能

优点：

  • 易于管理：可通过Docker CLI或API进行创建、删除等操作。
  • 安全性高：不暴露宿主机文件系统，减少安全隐患。
  • 跨平台兼容：支持Linux和Windows容器，便于多环境部署

缺点：

  • 路径固定：由Docker控制，不如绑定挂载灵活。

实例：

启动一个名为app03的Nginx容器，映射端口，并挂载两个目录——一个是网站根目录，另一个是Nginx配置目录，允许用户自定义和更新Nginx配置及网页内容。

面试题：目录挂载和卷映射有什么区别？

目录挂载（绑定挂载/Bind Mounts）和卷映射（数据卷/Volumes）的主要区别在于数据管理方式和初始化行为。

  • 目录挂载：直接映射宿主机上的指定路径到容器内，若宿主机路径为空，容器内挂载目录的内容会被隐藏，可能导致容器因缺少必要文件而报错（如403错误）。这种方式依赖宿主机的文件结构，适合开发环境快速修改代码，但安全性较低且跨平台兼容性差。
  • 卷映射：Docker自动管理存储路径（如/var/lib/docker/volumes/），若宿主机卷为空，容器启动时会自动将容器目录内的文件复制到卷中，确保运行所需文件的完整性 。数据卷独立于宿主机结构，支持跨容器共享，适合生产环境持久化数据，安全性更高且便于备份迁移。

Docker 网络

创建自定义网络，实现主机名作为稳定域名访问，可以轻松实现docker集群。

容器间通信配置

  • 案例1：容器间通信

容器app1可通过curl http://app2:80直接访问app2

总结：docker允许我们自己创建一个网络，以后的容器可以加入该网络，每一个容器的容器名就是网络访问的域名。所有通过Docker启动的容器在网络层面完全互通，容器之间可以任意互相访问，只需要使用容器IP或容器名加上目标服务的端口号进行通信，而无需外部暴露端口。

实战1：Redis主从同步集群

要求：实现在一个自定义的Docker网络内构建Redis集群，包括主节点和从节点的配置与启动，使它们能够通过内部网络进行通信和数据同步。

采用bitnami-redis镜像

网站：https://hub.docker.com/r/bitnami/redis

  • 第一步：docker network create mynet 创建一个名为 mynet 的自定义网络。

  • 第二步：

    • 主节点配置（redis01）:

      1. 启动第一个Redis容器作为主节点。

      2. 将主机的 /app/rd1 目录挂载到容器内的 /bitnami/redis/data，用于持久化数据；

      3. 设置环境变量以指定其为Redis主服务器并设定密码；

      4. 最后将其加入到之前创建的 mynet 网络中，并命名为 redis01。

    • 从节点配置（redis02）:
      1. 启动第二个Redis容器作为从节点
      2. 挂载数据目录、设置环境变量来指明其为主节点的从属关系及认证信息
      3. 加入相同的 mynet 网络，命名为 redis02。

Docker Compose 快速部署

Docker Compose 是 Docker 官方提供的用于定义和运行多容器应用的工具，通过一个 YAML 格式的配置文件（docker-compose.yml），用户可以轻松部署复杂的应用环境。

实验：搭建WordPress个人博客平台

老办法：命令式安装

这样的方式存在一种问题：假如我们要在一台新的电脑上重新部署WordPress，那么就需要记住之前搭建的命令再次运行，比较麻烦。

为了解决该问题，可以采用编写docker-compose.yml 文件，一次性把要启动的所有应用全部写好，在另一台机器部署时，只需要将该文件交给对方，在目标机器使用命令一键启动即可。

Docker Compose 特性

  • 增量更新

    • 修改 Docker Compose 文件，重新启动应用，只会触发修改项的重新启动。

  • 数据不删

    • 默认即使down了容器，所有挂载的卷并不会被移除，比较安全。
    • 可以采用docker compose down --help 查看参数，选择合适的参数将挂载的卷删除。
      • 例如：docker compose -f compose.yml down --rmi all -v 删除镜像和对应的卷。

Dockerfile—自定义镜像

Dockerfile 是一个文本文件，用于定义构建 Docker 镜像的步骤。它包含一系列指令和参数，指导 Docker 如何组装镜像。

常见指令

  • FROM: 指定基础镜像。
  • LABEL：自定义标签
  • RUN: 执行命令并创建新的镜像层。
  • COPY/ADD: 将本地文件复制到容器中。
  • EXPOSE: 容器运行时监听的端口。
  • CMD/ENTRYPOINT: 设置容器启动时默认执行的命令或程序入口点

更多指令可以参考官网：https://docs.docker.com/reference/dockerfile/

实例

为app.jar制作镜像

  1. 使用 rz 命令将jar包上传至服务器

  2. 编写Dockerfile文件：vim Dockerfile

  3. 使用 docker build -f Dockerfile -t MyjavaApp:v1.0 . 进行构建

    • 该命令的意思为：从当前目录下读取 Dockerfile 中定义的指令来构建一个新的 Docker 镜像，并将其命名为 MyjavaApp 版本 v1.0。

  4. 使用 docker run -d -p 8888:8080 MyjavaApp:v1.0 运行新构建的镜像

Docker 镜像分层存储机制

Docker 镜像分层存储机制就像搭建乐高积木一样，每一层都是一个独立的模块。当你需要构建一个新的镜像时，你可以选择不同的模块（即层）来组合成你想要的样子。

这些模块可以被重复使用，所以如果你有多个相似的项目，它们可以共享相同的模块，这样就节省了空间和时间。而且，因为每个模块都是独立的，所以在修改或更新某个部分时，只需要替换相应的模块即可，而不需要重新建造整个结构。

这种设计使得 Docker 镜像既高效又灵活，非常适合快速开发和部署应用。
