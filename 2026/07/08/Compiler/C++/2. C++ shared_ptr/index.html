C++ shared_ptr

2. C++ shared_ptr

引入

在神经网络里，经常有一种结构叫做“残差连接（Residual Connection）”

假设我们有一个图像输入的特征张量 Tensor X。在计算图中，它同时被另外三个算子节点依赖：

  1. Op A (卷积层 1) 需要读取 Tensor X
  2. Op B (卷积层 2) 需要读取 Tensor X
  3. Op C (加法节点) 也需要读取 Tensor X

这样就会引出一个问题：

  • 如果用裸指针：到底该由谁来 delete 掉 Tensor X？如果 Op A 执行完顺手把它删了，Op B 和 Op C 就会因为访问“悬垂指针”而直接导致整个 PyTorch 进程崩溃。如果大家都不删，1GB 的显存就泄漏了。
  • 如果用 unique_ptr：不好意思，unique_ptr 叫“独占所有权”。如果把 Tensor X 给了 Op A，Op B 和 Op C 连碰它的资格都没有。

面对这种**“一份数据，多人共享”**的复杂拓扑结构，我们急需一种全新的机制

技术破局 —— 引用计数与最后关灯法则

前人想出的解决思路极其优雅，生活里也极其常见，我称之为**“最后离开会议室的人负责关灯”**法则

我们需要一种**引用计数（Reference Counting）**机制：

  1. 初始状态：Tensor X 被创建出来，当前有 1 个人看着它（引用计数 = 1）
  2. 共享行为：Op A、Op B、Op C 都想用它，每一次有人引用它，就把数字加 1（引用计数 = 4）
  3. 离开行为：Op A 计算完了，不再需要 Tensor X，就把数字减 1（引用计数 = 3）
  4. 终极裁决：当最后一个依赖它的算子执行完毕，执行减 1 操作时，发现引用计数变成了 0。这就意味着大厅里空无一人了，这最后一个人（最后析构的 shared_ptr）负责顺手把 Tensor X 给 delete 掉（关灯）

假设 p1 活在主函数的栈上，它肚子里有 count_ = 1

接着，我们把 p1 拷贝给了 p2 (shared_ptr p2 = p1)。此时，就算我们在拷贝构造函数里写了让计数器加 1 的逻辑，结果会变成：

  • p1 肚子里的 count_ 变成了 2
  • p2 肚子里的 count_ 变成了 2

注意！这两个 count_ 是分别存在于两段完全不同的物理内存（栈内存）里的两个独立的整数！

如果此时，主函数里又创建了 p3 = p1。p1 和 p3 肚子里的计数变成了 3。但是 p2 根本不知道这件事！ p2 肚子里的数字还是可怜的 2

更可怕的是，当 p1 销毁时，它的计数减去了 1，但这完全影响不到 p2 和 p3。最终大家的计数彻底乱套，要么提前释放导致野指针，要么永远不释放导致内存泄漏

终极架构——引出“控制块 (Control Block)”

既然所有的 shared_ptr 都必须紧盯同一个计数器，那么这个计数器绝不能长在某一个智能指针的肚子里（栈上），它必须被独立出来，放在所有指针都能看到的地方（堆上）！

这就是 C++ 标准库给出的完美架构：

当你第一次创建一个 shared_ptr 时，它在底层其实做了两次内存分配（这也是为什么说 shared_ptr 开销比 unique_ptr 大的原因）：

  1. 申请一块堆内存，存你的真实数据（比如 Tensor）
  2. 额外申请一块极小的堆内存，叫做“控制块（Control Block）”，里面存着唯一的引用计数 count_

所以，一个真正的 std::shared_ptr 肚子里其实藏着两个裸指针：

这样一来，不管你克隆（Copy）出 100 个还是一万个 shared_ptr，大家的 control_block_ 指针都指向同一块内存。无论谁去修改计数，所有人看到的永远是最实时的全局状态

难点在于：在构造函数、拷贝构造、拷贝赋值、甚至析构函数里，同时小心翼翼地维护这两个指针！

骨架设计

  1. 底层变量（双指针架构）
    • 我们需要 T* ptr_（指向真实数据）和 size_t* ref_count_（指向堆上的控制块）
  2. 辅助函数 release_resource()
    • 在释放前，先把 ref_count_ 里的数字减 1。如果减到了 0，说明你是最后一个人，此时不仅要 delete ptr_（清理数据），还绝对不能忘记 delete ref_count_（清理计数器本身）！
  3. 显式构造函数
    • 第一次接管裸指针时，不仅要赋值 ptr_，还必须要在堆上 new 出一个整数作为控制块，并初始化为 1
  4. 析构函数
    • 不能无脑 delete 数据了，只能调用上面的 release_resource() 进行“试探性退场”
  5. 拷贝构造函数
    • 直接复制两个指针，并把堆上的计数器数字加 1
  6. 拷贝赋值运算符
    • 当执行 p1 = p2 时，p1 必须先放弃自己以前的资源（栈上的引用数减 1，如果为0还需要清理空间），然后再去复制 p2 的指针，并让 p2 会议室的人数加 1。注意千万别忘了防止自我赋值！

面试题

shared_ptr 的内部结构是怎样的？为什么说它比裸指针“重”？

参考回答：

一个标准的 shared_ptr 内部其实包含两个裸指针：

  • 指向真实的堆内存数据
  • 指向一个动态分配的控制块（Control Block）
    • 控制块里存放了引用计数、弱引用计数以及自定义删除器

因为它需要额外维护这个控制块，所以 sizeof(shared_ptr) 通常是裸指针的两倍（16字节）

而且在通过 new 构造时，需要进行两次堆内存分配（数据一次，控制块一次），这在极端性能敏感的算子开发中是一笔不可忽视的开销

shared_ptr 是线程安全的吗？（必考）

参考回答：

这是一个经典的陷阱，答案是部分线程安全

  1. 控制块是线程安全的：底层对引用计数的加减操作使用了原子操作，所以多个线程同时拷贝或销毁 shared_ptr 时，计数器绝不会错乱
  2. 内部指向的数据不是线程安全的：如果多个线程通过 shared_ptr 去读写它所管理的底层的 Tensor 数据，必须额外加锁

此外，在高度并发下，控制块的原子操作会导致缓存一致性风暴，也就是 Cache Line 争用，这也正是系统底层推荐使用 unique_ptr 替代 shared_ptr 的原因

如何优化 shared_ptr 的内存分配开销？

参考回答：

永远优先使用 std::make_shared 而不是 shared_ptr(new T)

因为 new T 会让真实数据和控制块分散在堆内存的两个不同地方，导致 CPU 缓存命中率（Cache Miss）下降

而 std::make_shared 会在底层进行一次内存融合，申请一块连续的大内存，把数据和控制块放在一起

这不仅减少了一次内存分配开销，还大幅提升了空间局部性，对性能更友好

shared_ptr 最致命的缺陷是什么？如何解决？

参考回答：

最致命的是循环引用（Circular Reference）

比如在构建复杂的计算图（AST）时，如果父节点拥有子节点的 shared_ptr，而子节点又反向拥有父节点的 shared_ptr，那么两者的引用计数永远无法降到 0，导致内存永久泄漏

解决方法：引入 std::weak_ptr。它像是 shared_ptr 的一个‘旁观者’，可以观测并访问数据，但绝对不会增加控制块里的引用计数。用 weak_ptr 打断循环，就能解决这个问题
