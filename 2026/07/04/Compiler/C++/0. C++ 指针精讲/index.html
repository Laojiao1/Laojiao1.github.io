C++ 指针精讲

0. C++ 指针精讲

Level 1：原生指针的物理本质与内存视角

1. 指针到底是什么？

在 64 位操作系统（比如你用的 x86_64 CPU 或现代 GPU）下，指针就是一个 64 位的无符号整数（8 个字节大小）。
不管这个指针是指向一个 1 字节的 char，还是指向一个占用几百 MB 的巨型 Tensor 结构体，指针本身永远只占 8 个字节。它里面存放的值，就是虚拟内存空间中的一个地址编号。

汇编层面：

在 CPU 看来，根本没有“指针”这个概念。当你执行 *p = 100 时，编译器生成的汇编指令大概是这样的：

mov DWORD PTR [rbx], 100 （假设指针的值存在 rbx 寄存器里，[rbx] 就代表去读取/写入这个地址指向的内存）。

解引用 (*p) 的本质，就是一次 CPU 的访存操作（Load/Store）。 在高性能计算中，最怕的就是指针乱指（Cache Miss），导致 CPU 苦等内存数据。

2. 既然都是 8 字节，为什么还要分 int*、float* 和 void*？

这是很多新手疑惑的地方：既然指针只是个地址，为什么要有类型？其实类型只服务于两件事：“解读步长” 和 “内存运算”。

① 解读步长（内存怎么切分）

内存是一片连续的字节数组（每个格子 1 字节）。当你拿到一个地址 0x1000 时：

  • 如果它是 char*，解引用时 CPU 只读 1 个字节。
  • 如果它是 float*，解引用时 CPU 会读 4 个字节，并按 IEEE 754 浮点数格式解码。
  • 如果它是 void*，它纯粹只是个地址，无法解引用，因为编译器不知道该读几个字节。（在手写底层内存池时，我们经常先分配一大块 void* 内存，再通过类型转换 reinterpret_cast 把它当成特定对象来用）。

② 指针算术（Pointer Arithmetic）—— 算子开发的核心

如果你对指针加一，它在内存中前进的距离取决于它指向类型的大小。

3. 悬垂指针 (Dangling Pointer) 与野指针 (Wild Pointer)

在造轮子时，这是导致 Segfault（段错误）的罪魁祸首。

  • 野指针：只声明了指针，但没有初始化（int* p;），它里面存着随机的垃圾地址。解引用直接崩溃。
  • 悬垂指针：指针指向的内存已经被释放了，但指针还在。

在 C++ 中，为了从根本上消灭这种悬垂指针，现代 C++ 才引入了 RAII（资源获取即初始化）和智能指针（这是我们后续要重点攻克的阶段）。

Level 2：指针常量和常量指针

1. 指针与数组的语法羁绊

在 C++ 语法中，数组名在很多情况下会退化（Decay）成一个指向它首元素的指针。

注意语法上的区别：sizeof(arr) 是整个数组的大小（20字节），而 sizeof(p) 永远是 8 字节（64位系统下的指针大小）。

2. const 与 指针

在 C++ 里，const 关键字和 * 的相对位置决定了到底是地址不能变，还是地址里面的内容不能变。

绝招：“从右向左读”（或者看 const 修饰的是谁）。

  • 语法 1：指向常量的指针 (Pointer to const)
  • 语法 2：常量指针 (Const pointer)
  • 语法 3：终极叠加 (指向常量的常量指针)

3. 函数传参语法：值传递 vs 指针传递

这是最基础也是最常用的语法。C++ 默认是“值传递”（Copy），如果要在函数里面修改外面的变量，就必须传指针。

总结

  1. int* const p：

    • 读法：从右向左读，p 是一个 const (常量)，它是一个 * (指针)，指向 int。
    • 本质：指针本身是常量。
    • 行为：p 里面存的地址不能变（不能再指别人），但可以通过 *p 去改那个地址里的值。

  2. const int* p (或 int const* p)：

    • 读法：从右向左读，p 是一个 * (指针)，指向一个 int 类型的 const (常量)。
    • 本质：指向常量的指针。
    • 行为：p 里面存的地址可以变（可以换个地方指），但绝对不能通过 *p 去修改那个地址里的值。

在很多国内的 C++ 教材、面试题甚至博客中，大家对 “常量指针” 和 “指针常量” 这两个词的叫法极其混乱，经常互相叫反，因此强烈建议彻底抛弃“常量指针”和“指针常量”这两个缩写词。

以后在心里直接叫它们的全称：

  • “指针本身是常量” ：int* const p
  • “指向常量的指针” ：const int* p

结合这两个全程和“从右向左看”的拆解法，这块语法就永远不可能再把你绕进去了

Level 3：new 和 delete 语法与内存泄露

1. 栈 (Stack) vs 堆 (Heap)

在聊语法前，必须先建立直觉：

  • 栈 (Stack)：你可以把它想象成“快餐店的流水线”。
    • 特点：极快、空间小（通常每个线程只有几 MB）、自动管理。
    • 语法表现：你平时写的 int x = 10; 或者 float arr[100]; 全在栈上。函数一结束，这块内存立刻被系统自动收回。
  • 堆 (Heap)：你可以把它想象成“巨大的公共仓库”。
    • 特点：极大（可以用尽系统的几十 GB 内存）、稍微慢点、全手动管理。
    • 语法表现：当你在运行期间，突然发现你需要一个巨大的 Tensor（比如 100MB），栈根本放不下，或者这个 Tensor 必须在函数结束后还要继续存活，你就必须去堆里借空间。

2. new 和 delete 的基本语法

在 C++ 中，向堆（操作系统）借内存的语法是 new，还回去的语法是 delete。

3. 内存泄漏（Memory Leak）是怎么发生的？

在 C/C++ 里，操作系统对“堆”的态度是：你不主动 delete，我就永远不收回，哪怕你的指针变量已经死了。

这就好比：你租了一个仓库，把唯一的钥匙（指针）给丢了，这仓库你进不去，房东（OS）也以为你还在用，不敢租给别人。

这就是内存泄漏。程序跑得久了，内存会被吃光，然后被系统无情地 Kill 掉（OOM）。

4. 为什么要引入这些

你会发现，每次调用 new 和 delete，底层其实都要和操作系统打交道（去遍历哪里有空闲内存）。这是一个非常慢的过程。

所以在高性能计算、手写 Thread Pool 或者 LRU Cache 时，我们绝对不会在运算的核心循环里去 new 和 delete。

我们通常的做法是：在一开始 new 出一整块巨大的内存（这就是 Memory Pool 的雏形），然后自己写逻辑用指针在这块大内存里切分、重用。

Level 4：RAII 与智能指针的雏形

1. 核心机制：析构函数语法与 RAII

在 Level 3 里我们说过，活在栈（Stack）上的变量，在函数结束时会被系统自动回收。

C++ 语言的创造者利用了这一点，发明了一个极其聪明的语法机制：析构函数 (Destructor)。

这就是 C++ 中最伟大的发明之一，叫 RAII（资源获取即初始化）。

它的语法本质是：把需要手动 delete 的堆内存，包装进一个栈上的对象里。利用栈对象生命周期结束时自动调用析构函数的语法特性，来替我们执行 delete。

现代 C++（C++11 开始）的标准库已经为你提供好了现成的“Box”，就是我们常说的智能指针。你以后写代码，基本不再需要手动写 new 和 delete 了。

2. std::unique_ptr (独占式指针) 的基础语法

这是你以后写代码最常用、性能最好（零额外开销）的指针。它表示**“这段堆内存，只有我一个人拥有”**。

3. std::shared_ptr (共享式指针) 的基础语法

在写多线程（比如把一个大矩阵传给好几个线程并行计算）时，这段内存是共享的，不能被某一个线程独占。这时候就用 shared_ptr。

一些琐碎的知识点

补充 1：引用 (&) 的底层其实就是指针

在 C++ 里，我们在函数传参时，经常纠结到底是传指针 void func(int* p) 还是传引用 void func(int& r)。
语法真相：在编译器（汇编）眼里，引用就是指针。
int& r = x; 在底层完全等价于上一关我们讲的常量指针 int* const r = &x;。

语法的区别在于：

  1. 引用绝对安全：指针可以为空 (p = nullptr)，但引用在声明时必须初始化，且永远不能为空。
  2. 语法糖更甜：使用引用时，不需要写繁琐的解引用符号 *p 和取地址符号 &，直接像用普通变量一样用 r = 100; 即可。

开发建议：在 C++ 中，能用引用传参的地方，绝对不用指针（除非这个参数有可能传个空值进来表示特殊含义）。

补充 2：类里面幽灵般的 this 指针

当你在手写数据结构（比如 LRU Cache 的 Node 节点）时，你一定会写类（Class）。

只要你在类的非静态成员函数里，编译器就会悄悄塞给你一个隐式指针，名字就叫 this。它的类型是 Node* const this（当前对象的常量指针），专门用来解决成员变量和参数重名的问题。

总结

到这里，C++ 指针的基础语法体系就完整了：

  1. 原生指针 (*)：知道它是装地址的 8 字节整数，注意 const 的位置。
  2. 堆内存 (new/delete)：懂得了手动向系统借内存和还内存的语法（特别注意数组的 delete[]）。
  3. 现代指针 (make_unique/make_shared)：利用析构函数（RAII 机制），让编译器替我们自动执行 delete。

以后如果要写 AI 算子和架构代码，原生的 new/delete 尽量不要碰了，但原生指针 int* 还是会大量用到（比如 unique_ptr::get() 可以把底层原生指针取出来传给 C 语言写的底层 CUDA 接口）
