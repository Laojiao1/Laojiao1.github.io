神经网络量化 —— Scale 注入模型实战

在工业界（尤其是对接 TensorRT、SNPE 等后端时），直接导出一个带有 Q/DQ（QuantizeLinear / DequantizeLinear） 节点的 ONNX 模型是最标准的做法。

这样可以把我们在 Python 侧精心校准好的 Scale 和 Zero-Point 强行固化在模型结构里，不给推理引擎任何胡乱解释的机会。

今天我们用一个最干净、最闭环的实战代码，来实现这个完整的流水线

实战目标

  1. 构建一个包含 Conv2d 和 ReLU 的简单浮点模型。
  2. 模拟前向传播，利用白盒方式拿到这一层的最佳 Scale（这里直接用我们上一关算出的黄金 Scale）。
  3. 使用 PyTorch 2.0+ 推荐的白盒量化工具（torch.ao.quantization） 或是 ONNX 官方工具链，显式地将 Q/DQ 节点“拍”进网络。
  4. 导出 ONNX 并教你如何用 Netron 检查它。

总的来说，这张图可以分成三大核心区来拆解：左侧输入量化区、右侧权重量化区，以及中下部的激活量化与输出区。

左侧输入量化（激活值 Fake Quantize）

从最上方的 input 开始，数据向下流动。原本在浮点模型里，input 应该直接连到 Conv（卷积）算子上。但现在，中间被强行插入了一套由 6 个算子组成的“伪量化组件”：

这 6 个算子分为两半，完美对应了量化与反量化数学公式

1. 前半段（量化）：Div -> Round -> Add -> Clip

这四个算子在执行：q=clamp(round(x/S)+Z,−128,127)q = \text{clamp}(\text{round}(x / S) + Z, -128, 127)q=clamp(round(x/S)+Z,−128,127)

  • Div (除法)：输入的浮点张量（xxx）在这里除以 B <1>（也就是你算出来的输入 scale = 0.007812）。
  • Round (四舍五入)：将缩放后的浮点数就近舍入到最接近的整数。
  • Add (加法)：加上零点 ZZZ（因为是对称量化，这里的 B <1> 常量值为 0，加 0 后数值不变）。
  • Clip (截断)：图上清清楚楚写着 min = -128, max = 127。凡是超出这个范围的离群值（Outliers），都会被在这里硬生生“砍一刀”，按死在边界上，防止硬件计算溢出。
  • 至此，数据在概念上变成了真正的 INT8 整数。

2. 后半段（反量化）：Sub -> Mul

这两个算子在执行：r=(q−Z)×Sr = (q - Z) \times Sr=(q−Z)×S

  • Sub (减法)：减去零点 ZZZ（值仍为 0）。这里有一个有意思的细节，你注意到 Sub 的左边分叉连着一个 Identity 节点了吗？因为在代码中零点被强行初始化并广播成了 Tensor，Identity 只是一个占位标识，确保算子输入流的拓扑结构完整。
  • Mul (乘法)：将整数乘以刚才的 scale（B <1> 即 0.007812）。
  • 至此，数据又变回了浮点数（FP32），并流向 Conv。

右侧权重量化

在 Conv 算子的右侧，是针对卷积核权重（conv.weight）的伪量化路径。

  • 它的算子顺序稍微有些不同：Identity (权重本身) 和 Round 分头行动，经过 Add、Clip、Sub、Mul。
  • 为什么少了一个 Div？：
    • 因为权重的 scale 是在导出时就已经确定并且固定的。PyTorch 在导出 ONNX 时，为了优化计算图，提前在离线状态下把权重除以 Scale 的步骤做完了。所以留在图里的 Round 已经是缩放后的结果，后面只需要做 Clip（确保在 -128 到 127 之间）和 Mul（反量化乘以权重的黄金 scale = 0.004567）即可。

核心计算与输出量化（Conv + ReLU + Output Quant）

当左边的输入（激活值）和右边的输入（权重）都在浮点空间完成了“量化再反量化”的仿真后，它们终于在 Conv (蓝色算子) 汇合，执行矩阵乘加运算。

  1. 算子融合的体现：之前提到过：“框架是在融合前做量化统计，还是融合后做？这两种直方图算出来的 Scale 差了十万八千里”。在我们的白盒代码里，我们显式执行了 fuse_modules。可以看到，Conv 后面紧跟着 Relu，而输出激活值的伪量化组件（Div →\rightarrow→ Round … →\rightarrow→ Mul）被精准地安置在 Relu 的出口下方。这确保了我们统计直方图、计算 KL/MSE 时的输入数据，全都是经过 ReLU 过滤后的真实分布，实现了完美的白盒对齐。
  2. 最后输出：经过 ReLU 之后的特征图（Feature Map），再次经过最后一组组件的约束，乘以输出 scale = 0.012345。最终从 output 吐出（尺寸为 1×16×224×2241 \times 16 \times 224 \times 2241×16×224×224）。

思考：这密密麻麻的算子，怎么加速？

你看到这张图可能会想：“天哪，原本只要跑一个 Conv 和 ReLU，现在前后加起来多了十几个 Div、Round、Mul、Clip 算子，这模型不应该变得巨慢无比吗？”

这就是量化对齐（Quantization Alignment）最神奇的地方：

当你把这个 ONNX 文件提交给 TensorRT、ONNX Runtime（带有精度加速后端）或者车端芯片的 NPU 编译器时，编译器内部的优化器（Graph Optimizer）会玩一个叫 “模式匹配（Pattern Matching）” 的游戏：

  • 它一旦抓取到 [Div -> Round -> Add -> Clip -> Sub -> Mul] 这个特定的组合，就会发出警报：“报告！发现标准伪量化节点！”
  • 编译器会无情地把这些多出来的数学算子全部从图里抹去（强力擦除）。
  • 它会把这些算子身上携带的 scale 数字（0.007812、0.004567 等）直接抽出来，作为配置参数固化到硬件的硬件寄存器中。
  • 在底层芯片真正执行的那一瞬间，这个复杂的图在硬件里会坍塌成这个样子：

真实的硬件流水线变成了：

纯 INT8 输入⟶NPU/Tensor Core 内部高效执行整数卷积运算⟶纯 INT8 输出\text{纯 INT8 输入} \longrightarrow \text{NPU/Tensor Core 内部高效执行整数卷积运算} \longrightarrow \text{纯 INT8 输出} 纯 INT8 输入⟶NPU/Tensor Core 内部高效执行整数卷积运算⟶纯 INT8 输出

所以现在的这张图，是“用 FP32 算子做完美的 INT8 行为仿真”。它在格式上完全锁死了数值分辨率，而在实际部署运行的时侯，它将享受纯整数流水线带来的翻倍吞吐量和极低的 Latency。

看到这里你可能会有一个疑问：

既然这叫量化模型，为什么送进 Conv 算子的依然是经过 Mul（反量化）出来的浮点数？这不还是浮点计算吗？

然而这恰恰是工业级 QDQ 模型的高明之处，主要有以下三个核心原因：

浮点框架（PyTorch/ONNX Runtime）在运行期需要“数值仿真”

在 Python 侧或标准的 ONNX 运行时，框架底层的算子（例如 conv2d）在默认情况下都是 FP32（单精度浮点）

  • 如果直接送入 INT8 整数：浮点卷积算子根本无法解析定点整数，或者会把整数 127 直接当成浮点数 127.0 去计算，整个网络的数值流直接就彻底崩塌了。
  • 反量化的真实意图：通过 Mul（乘以 Scale），把数据拉回浮点空间（FP32），但是此时的浮点数已经失去了原本高精度的连续性，变成了断续的、“带有阶梯感”的浮点数。

也就是说，反量化后的 FP32 数值在框架里运行，能够完美仿真出模型在 INT8 芯片上运行时所产生的精度损失和噪声。这就叫“模拟量化（Fake Quantization）”。

算子融合（Operator Fusion）的“接头暗号”

这在工程落地上被称为 “模式匹配（Pattern Matching）”。

推理引擎（如 TensorRT、芯片 NPU 编译器）在读取 ONNX 模型时，它不是傻傻地一个算子一个算子去执行的，它会进行大规模的计算图优化。

当编译器看到一整串 [Div -> Round -> Add -> Clip -> Sub -> Mul] -> Conv 时，它就会识别出这是一个标准的 QDQ（Quantize-Dequantize）模式。编译器会把这个结构看作是一个“接头暗号”：

  • 编译器的动作：它会把这一堆模拟量化的浮点算子（Div, Round, Mul 等）彻底从计算图里抹去，物理擦除。
  • 硬件的替换：它会把这些算子里自带的 scale 提取出来，直接配置到硬件（如 Tensor Core 或 DSP）的寄存器中。
  • 运行时的真相：在实际芯片运行的一瞬间，硬件会直接调用纯定点计算指令（如 vpmaddubsw），直接执行纯 INT8 的矩阵乘加。

所以，留在 ONNX 里的“反量化”是为了在离线状态下把 Scale 参数带给编译器。一旦编译结束，反量化在运行时不占用任何计算开销。

为了 QAT（量化感知训练）中反向传播的梯度流动

如果你是在训练阶段（QAT）加入这个组件，反量化更是不可或缺：

  • 如果你把特征直接变成了真正的 INT8 整数，在反向传播计算梯度时，整数的梯度是离散的，甚至 Round（四舍五入）算子在数学上的导数处处为 0（或者是无穷大）。这会导致反向传播的梯度在这一层彻底断绝，模型根本无法训练。
  • 通过反量化转回 FP32，梯度就可以沿着连续的浮点数一路顺畅地反向传导回前面的层，让权重可以继续通过梯度下降进行微调，从而主动去学出一套能够适应 INT8 噪声的权重。
