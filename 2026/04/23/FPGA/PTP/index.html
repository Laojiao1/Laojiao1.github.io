PTP 协议

PTP 协议

精确时间协议（Precision Time Protocol，简称 [PTP][https://support.huawei.com/enterprise/zh/doc/EDOC1100174722/7631975b]）

引入

为什么需要 PTP？（一个生活中的类比）

假设你和几个朋友要在不同的大楼里同时按下快门拍摄一场流星雨。

  • 方法 A： 大家都看路边的公共大钟。但钟可能有快有慢，误差几秒钟，拍出来的照片对不上。
  • 方法 B： 你们用手机对时。网络稍微一卡，手机显示的秒数就差了几十毫秒（0.01秒）。对于高速运动的物体，这依然不够准。
  • PTP 的目标： 让所有设备的时间差缩小到纳秒级（10−910^{-9}10−9 秒）。这就像是让分布在全世界的快门，在同一亿分之一秒内同步按下。

PTP 的核心逻辑：主从关系

在 PTP 网络中，大家不是平等的，而是有等级的：

  • Grandmaster (GM) - 主时钟： 它是整个网络的“时间源头”，通常连接了 GPS 卫星或原子钟，它说现在几点，全网就得信它。
  • Slave (Ordinary Clock) - 从时钟： 它们盯着主时钟，不断调整自己的表，确保跟主时钟一模一样。

PTP 是如何做到“超高精度”的？（核心原理）

这是 PTP 最牛的地方。它不仅仅是“发个报时短信”，它还要计算短信在路上走的时间。

我们可以把这个过程简化为**“四次盖章”**：

  1. 第一步（发送）： 主时钟给从时钟发一个包（Sync），并记录下发出的瞬间时间 t1t_1t1​。
  2. 第二步（接收）： 从时钟收到这个包，记录下收到的瞬间时间 t2t_2t2​。
  3. 第三步（反馈）： 从时钟再给主时钟发一个包（Delay_Req），记录发出时间 t3t_3t3​。
  4. 第四步（确认）： 主时钟收到后，记录下收到时间 t4t_4t4​，并发回给从时钟。

现在，从时钟手里有 4 个时间点。通过简单的数学公式，它就能算出来：

  • 路径延迟（路上花了多久）： Delay=(t4−t1)−(t3−t2)2Delay = \frac{(t_4 - t_1) - (t_3 - t_2)}{2}Delay=2(t4​−t1​)−(t3​−t2​)​
  • 时间偏差（我的表比你快/慢了多少）： Offset=(t2−t1)−DelayOffset = (t_2 - t_1) - DelayOffset=(t2​−t1​)−Delay

敲黑板： PTP 厉害就在于它考虑了网络传输造成的延迟，并把它减掉了！

这四个时间点是怎么计算的?

在 PTP 协议中，为了算出从钟比主钟慢了多少，必须解决一个难题：信息在网线和交换机里跑，到底花了多少时间？

为了搞清楚这一点，PTP 设计了一个“往返跑”的过程，一共产生 4 个关键的时间戳（Timestamp）：t1,t2,t3,t4t_1, t_2, t_3, t_4t1​,t2​,t3​,t4​。

1. 四个时间戳的产生过程

我们假设 Master 是主钟，Slave 是从钟。

第一阶段：测量“下行”时间

  1. t1t_1t1​ (Sync 消息发送时间)： Master 发出一个叫做 Sync 的同步报文。在报文离开 Master 网卡的瞬间，硬件记下精确时间 t1t_1t1​。
  2. t2t_2t2​ (Sync 消息接收时间)： Slave 收到这个 Sync 报文。在报文到达 Slave 网卡的瞬间，硬件记下精确时间 t2t_2t2​。
    • 此时 Slave 知道：t2=t1+延迟+偏差t_2 = t_1 + \text{延迟} + \text{偏差}t2​=t1​+延迟+偏差。但它还不知道具体的延迟是多少。

第二阶段：测量“上行”时间

  1. t3t_3t3​ (Delay_Req 发送时间)： Slave 想要测算往返时间，于是主动发一个 Delay_Req（延迟请求）给 Master，并记下出发瞬间的时间 t3t_3t3​。
  2. t4t_4t4​ (Delay_Req 接收时间)： Master 收到这个请求，记下到达瞬间的时间 t4t_4t4​，然后通过一个 Delay_Resp（延迟响应）消息把 t4t_4t4​ 告诉 Slave。

2. 数学时间：怎么算出来的？

现在 Slave 手里集齐了 t1,t2,t3,t4t_1, t_2, t_3, t_4t1​,t2​,t3​,t4​

A. 计算单向延迟 (Mean Path Delay)

假设报文“过去”和“回来”的时间是一样长的（这叫对称性假设）。

那么，往返总共花的时间就是：

(t4−t1)−(t3−t2)(t_4 - t_1) - (t_3 - t_2) (t4​−t1​)−(t3​−t2​)

解读：(t4−t1)(t_4 - t_1)(t4​−t1​) 是总耗时，(t3−t2)(t_3 - t_2)(t3​−t2​) 是 Slave 内部处理等待的时间。

所以，单向延迟 (Delay) 为：

Delay=(t4−t1)−(t3−t2)2Delay = \frac{(t_4 - t_1) - (t_3 - t_2)}{2} Delay=2(t4​−t1​)−(t3​−t2​)​

B. 计算时间偏差 (Clock Offset)

知道了路上花了多久，Slave 就能算出自己的表到底歪了多少。

如果 Slave 的表是准的，那么收到消息的时间 t2t_2t2​ 应该等于 t1+Delayt_1 + Delayt1​+Delay。

所以，偏差 (Offset) 就是：

Offset=t2−t1−DelayOffset = t_2 - t_1 - Delay Offset=t2​−t1​−Delay

结论： Slave 只要把自己的时钟减去这个 OffsetOffsetOffset，就实现了和 Master 的完美同步！

3. 一个直观的例子（带入数字）

为了彻底明白，我们带入一组容易理解的数字：

  1. Master 在 10:00:00 (t1t_1t1​) 发出信息。
  2. Slave 在自己的表显示 10:00:10 (t2t_2t2​) 时收到。 (这时 Slave 看起来比 Master 快 10 秒，但别忘了路上还要时间)。
  3. Slave 在 10:00:20 (t3t_3t3​) 发回请求。
  4. Master 在 10:00:12 (t4t_4t4​) 收到。

我们来算算：

  • 往返总耗时： (12−0)−(20−10)=12−10=2(12 - 0) - (20 - 10) = 12 - 10 = 2(12−0)−(20−10)=12−10=2 秒。
  • 单向延迟 (Delay)： 2/2=12 / 2 = 12/2=1 秒。（说明信息在路上跑了 1 秒）
  • 偏差 (Offset)： 10−0−1=910 - 0 - 1 = 910−0−1=9 秒。

结果： Slave 发现自己的表其实比 Master 快了 9 秒。于是它把表往回拨 9 秒，对时完成！

4. 为什么这比想象中难？

在现实中，这 4 个时间戳是纳秒级的。

  • 难点 1： 报文在交换机里排队怎么办？（PTP 交换机有专门的补偿机制）。
  • 难点 2： 如果去程和回程的路径不一样长（不对称），计算就会有误差。这是 PTP 调优中最头疼的问题。

为什么它比普通的 NTP（网络时间协议）准？

你可能听过 NTP（电脑右下角自动对时用的）。PTP 比 NTP 准一千倍以上，原因主要有两点：

  1. 硬件打戳 (Hardware Timestamping)： NTP 是靠软件（操作系统）记时间的，软件慢一点、CPU 忙一点，时间就不准了。PTP 直接在网卡硬件上记时间，像扣动扳机的瞬间直接记录。
  2. 网络层级更扁平： PTP 专门为局域网优化，在专门的交换机支持下，误差极小。

PTP 都在哪儿用？

  • 5G 基站： 基站之间必须严格同步，否则你走在街上手机信号切换时就会断网。
  • 自动驾驶： 多个摄像头和雷达的数据必须在同一时间点对齐，否则系统判断不出物体的准确位置。
  • 电力电网： 监控电流波动，必须精确知道故障发生的先后顺序。
  • 金融交易： 毫秒级的下单顺序决定了谁能赚到钱。

一个思考：

在刚才的笔记中，我提到了 t1t_1t1​（主钟发出的瞬间）。但这里有一个**“先有鸡还是先有蛋”**的硬件难题：我还没发出去，你是怎么知道我发出去的时间？

主钟在准备 Sync 报文时，报文内容里需要填上 t1t_1t1​。

  • 但是，当 CPU 正在写报文内容的时候，报文还没发出去呢！
  • 等报文真正离开网卡、产生 t1t_1t1​ 的那一刻，报文内容已经封死了，改不了了。

为了解决这个“预知未来”的问题，PTP 设计了两种方案：

1. 两步法 (Two-Step) —— “发个短信补一下”

这是最常用的稳妥方案。

  • 第一步： Master 发送 Sync 报文，里面 t1t_1t1​ 的位置先填个 0。当报文离开时，硬件记下真实的 t1=10:00:01.001t_1 = 10:00:01.001t1​=10:00:01.001。
  • 第二步： Master 紧接着再发一个报文，叫 Follow_Up，里面写着：“嘿，刚才那个包发出的准确时间是 10:00:01.00110:00:01.00110:00:01.001”。
  • Slave： 收到两个包后，把它们拼在一起，就知道 t1t_1t1​ 是多少了。

2. 一步法 (One-Step) —— “边跑边改”

这需要更高级、更昂贵的硬件支持。

  • 操作： 硬件在 Sync 报文经过网卡出口的那一纳秒，直接动态地把当前的精准时间戳“塞”进正在移动的数据流里。
  • 这就像是在飞驰的火车上，有人在车厢外瞬间刷上了出发时间。这种方法效率最高，但对交换机和网卡的要求极高。

注意：在实际配设备的时候，如果把主钟设成了“两步法”，而从钟只支持“一步法”，它们可能就对不上表或者报错，这是很容易踩的一个坑。

预备知识

1. PTP 的“时空结界”：Domain（域）

在一个物理网络里，可能同时跑着好几套 PTP 系统。比如：

  • 5G 基站用一套 PTP。

  • 旁边的电力监控用另一套 PTP。

    • 为了互不干扰，PTP 引入了 Domain Number（域号）。

  • 只有 Domain Number 相同的设备，才会互相“选老大”和同步。

  • 默认通常是 0。你可以理解为这是 PTP 的“频道”，频道不对，哪怕网线连着也对不上表。

2. 两种延时测量机制：E2E vs P2P (正片升级版)

之前提了一嘴，这里我们需要深入一点。虽然都是算 t1t_1t1​ 到 t4t_4t4​，但逻辑上有本质区别：

E2E (End-to-End, 端到端)
  • 做法： 从钟直接发 Delay_Req 给主钟，测量全路径的延迟。
  • 特点： 简单粗暴。中间可以经过不支持 PTP 的普通交换机（虽然精度会掉）。
  • 适用： 拓扑经常变化的场景。
P2P (Peer-to-Peer, 点对点)
  • 做法： 不测全路径。每一对相邻设备（主从、主换、换换）都会不断测量彼此的**“邻居延迟”**（使用 Pdelay_Req 报文）。
  • 特点： 极高精度。要求路径上所有设备必须支持 PTP。
  • 优势： 无论网络多复杂，延时是“分段叠加”的。如果主备切换，同步速度极快。

3. PTP 协议族：Profile（配置规范）

PTP 就像一套乐高，有很多零件。不同的行业为了方便，把需要的零件打包成一个“套餐”，这就是 Profile。

你买设备时，必须确认 Profile 是否匹配：

  • IEEE 1588 Default Profile： 最基础的。
  • IEEE 802.1AS (gPTP)： 汽车电子和音视频（AVB）专用，只支持 P2P 和一步法。
  • SMPTE ST 2059-2： 广播电视系统专用。
  • Telecom Profile (G.8275.1)： 电信运营商 5G 专用。

4. 这里的“度量衡”：TAI vs UTC

这块虽然冷门，但很关键。

  • UTC（协调世界时）： 也就是我们手机上的时间，它有**“闰秒”**。

  • TAI（国际原子时）： 严格按照原子振荡频率走的，没有闰秒。

    PTP 内部跑的是 TAI 时钟。 为什么？因为如果某天突然多出 1 毫秒的“闰秒”，对于纳秒级的 PTP 来说，整个系统的数学逻辑会瞬间崩溃。所以 PTP 先按 TAI 跑，最后在显示给用户看时，再加上一个 Offset（目前 TAI 比 UTC 快了 37 秒）。

BMCA 算法：谁是“话事人”？

在 PTP 网络里，不能有两个“老大”（Master），否则大家的时间就乱了。PTP 有一套极其严密的自动竞选机制，叫 最佳主时钟算法（Best Master Clock Algorithm, BMCA）。

每个设备都会不断发出 Announce（公告） 报文，里面晒出自家的“底牌”。大家拿到报文后，会按照以下顺序“比大小”，输的人自动变 Slave：

  1. Priority 1（优先级1）： 这是管理员手动设定的，0 最优先，255 最垃圾。
  2. Clock Class（时钟等级）： 硬件实力。比如连接了原子钟的设备等级最高，GPS 次之。
  3. Clock Accuracy（精度）： 你的表准到什么程度？纳秒级还是微秒级？
  4. Offset Scaled Log Variance（稳定性）： 你的表走得稳不稳？会不会一会儿快一会儿慢？
  5. Priority 2（优先级2）： 如果上面都一样，给管理员第二次手动干预的机会。
  6. MAC 地址： 最后的绝杀，谁的 MAC 地址数值小，谁赢。

三报文

虽然 PTP 报文很多，但你只需要记住这三类：

  1. 事件报文 (Event Messages)： 必须打精准时间戳的（Sync, Delay_Req）。它们是**“干苦力的”**。
  2. 一般报文 (General Messages)： 传递信息的，不用打戳（Follow_Up, Announce, Delay_Resp）。它们是**“传话的”**。
  3. 管理报文： 配置参数用的。

三时钟

这是 PTP 区分于 NTP 的技术杀手锏。普通交换机对时钟报文是“一视同仁”的，这会导致报文在交换机里排队，产生不确定的延迟（抖动）。为了消灭交换机带来的“幽灵延迟”，PTP 规定了两种高级设备形态：

  1. OC (Ordinary Clock，普通时钟) —— 它是网络的“终点”或“起点”。
  • 特点： 它只有一个 PTP 端口。
  • 角色： 如果它连接了 GPS 卫星，它就是 Grandmaster (GM)，全网的源头。
    • 如果它是一个传感器、摄像头或者基站，它就是 Slave，负责接收时间。
  • 比喻： 就像是一个“终端用户”，要么是发号施令的司令官，要么是听令的士兵。
  1. BC（Boundary Clock，边界时钟）—— 它是网络的“中继站/分销商”。
  • 特点： 它有多个端口。
  • 逻辑： 它的一个端口是对外的“耳朵”（Slave 端口），听老大的；其他端口是“嘴巴”（Master 端口），给下级传话。
  • 作用： 解决层级问题。如果全网几千个从钟都直接找 Grandmaster 对时，Grandmaster 会被活活累死（处理不过来那么多请求）。BC 分担了压力，并在本地重新修正了时间。
  1. TC（Transparent Clock，透明时钟）—— 它不参与对时，它像是一个“自带秒表的质检员”。
  • 过程： 报文进入交换机那一刻，它掐表；报文离开时，它停表。
  • 操作： 它计算出报文在自己肚子里待了多久（比如 50.5 微秒），然后直接修改报文里的一个特殊字段——Correction Field（修正域）。
  • 结果： 终端收到报文后，减去这个修正值，交换机内部的延迟就彻底“透明”消失了！

总结：三者如何配合？

想象一个大型工厂：

  1. OC (Grandmaster)： 房顶上的 GPS 接收机，它是时间的“唯一真理”。
  2. BC / TC： 厂房里的工业交换机。
    • 用 BC 就像是在每个车间设了一个“副钟”，车间内的设备都跟副钟对齐。
    • 用 TC 就像是换了一根“神笔网线”，报文穿过它时，它自动补齐了路上的延时。
  3. OC (Slave)： 生产线上的机械臂，它们接收 BC 或 TC 传来的信号，确保动作一致。

两种测量模式：E2E vs P2P

你在配置 PTP 设备时，必须二选一。选错了，系统直接罢工。

模式 全称 原理 适用场景
E2E End-to-End 终端直接给主钟发请求，测全路径总延迟。 简单网络，或者中间有普通交换机。
P2P Peer-to-Peer 每一对相邻的设备互相测“邻居间距离”。 只有支持 PTP 的设备组成的严密网络（如工业自动化）。

一个比喻：

  • E2E 是问“从我家到公司要多久”；
  • P2P 是问“我家到路口多久，路口到公司多久”。

报文格式

在 Wireshark 抓包时， PTP 报文就像一个剥开的洋葱，层层嵌套。

所有的 PTP 报文（不管是 Sync 还是 Announce）都共享一套公共头部（Header），然后再根据报文类型挂载不同的正文（Body）。

1. PTP 报文的“三层外壳”

首先，PTP 通常是跑在以太网上的，它的封装结构如下：

  • 以太网层： 规定了 MAC 地址。
  • 网络/传输层： 通常使用 UDP 协议（端口 319 用于高精度事件报文，320 用于普通报文）。
  • PTP 协议层： 包含 Header、Body 和可选的 Suffix。

2. 核心：公共报文头 (Message Header)

所有 PTP 报文的前 34 个字节 是完全一样的。这非常关键，因为交换机（TC/BC）就是靠读这 34 个字节来快速判断该如何处理这个包。

最重要的几个字段：

  • messageType (4 bits): 报文的身份。
    • 0x0: Sync（同步）
    • 0x8: Follow_Up（后续）
    • 0xB: Announce（公告）
  • versionPTP (4 bits): 协议版本。现在基本都是 2 (IEEE 1588-2008) 或 2.1。
  • domainNumber (8 bits): 域号。就像我之前说的“频道”，只有频道对上才能同步。
  • flagField (16 bits): 标志位。这里会标明是“一步法”还是“两步法”。
  • correctionField (64 bits): 最值钱的字段！ 这是一个 8 字节的纳秒计数器。TC 交换机在经过时会修改这个值，补上路上的延时误差。
  • sourcePortIdentity (10 bytes): 谁发的这个包？包含发件人的时钟 ID（类似 MAC 地址）和端口号。
  • sequenceId (16 bits): 序号。用来匹配 Sync 和 Follow_Up，或者 Delay_Req 和 Delay_Resp。

3. 报文正文 (Message Body)

根据 messageType 的不同，Header 后面跟着的内容也不同。我们看两个最典型的：

① Announce 报文（选老大用的）

它的 Body 里装满了“竞选筹码”：

  • Grandmaster Priority 1 & 2：手动设定的优先级。
  • Grandmaster Clock Class：时钟等级。
  • Steps Removed：离顶级主钟经过了几跳交换机。
② Sync / Delay_Req 报文（对时用的）
  • Origin Timestamp (10 bytes)：这就是传说中的时间戳。
    • 注意： 在两步法里，这个字段通常是 0，真实的时间戳在后面的 Follow_Up 报文里；在一步法里，这里就是最精准的发射时间。

4. 一个实战小细节：UDP 端口 319 vs 320

你可能会好奇，为什么要分两个 UDP 端口？

  • UDP 319 (Event Message)： 这里的报文（如 Sync, Delay_Req）在离开网卡的一瞬间，硬件必须**“咔嚓”一下打上时间戳**。操作系统会给这个端口最高优先级的处理权限。
  • UDP 320 (General Message)： 这里的报文（如 Announce, Follow_Up）只是传个话，不需要硬件精准打戳，走普通流程就行。

5. 总结：如何识别一个 PTP 包？

当你抓到一个包，看这三个地方就能认清它：

  1. 看端口： 319 还是 320？
  2. 看偏移量 0 的位置： 获取 messageType，确定它是 Sync 还是其他的。
  3. 看偏移量 8 的位置： 读 correctionField，看中间交换机有没有帮它修正时间。

掌握了报文格式，你就可以在 Wireshark 里手动分析 PTP 的对时过程了。

抓包实战：你在 Wireshark 里会看到什么？

当你打开抓包软件，PTP（协议显示为 1588）的流量主要分为两波：

  1. UDP 319 端口： 负责传带有“精准时间戳”的消息（事件报文，如 Sync）。
  2. UDP 320 端口： 负责传普通的“管理/后续”消息（通用报文，如 Announce, Follow_Up）。

提示： 如果你发现只有 320 没有 319，那说明你的硬件打戳功能没开，或者防火墙把关键的对时包拦住了。

下面我们来看一个典型的**两步法（Two-Step）**下，主钟发送 Sync 报文的真实样子。

1. 抓包快照：Sync 报文 (UDP 端口 319)

当你在 Wireshark 里点开一个 Sync 报文，你会看到如下层级结构：

Ethernet II 层
  • Destination: 01:1b:19:00:00:00 (这是 PTP 的多播 MAC 地址)
  • Type: IPv4 (0x0800)
User Datagram Protocol (UDP) 层
  • Source Port: 319
  • Destination Port: 319
Precision Time Protocol (PTP) 层

这就是我们分析的核心：

字段名称 典型值 含义
messageType 0x0 (Sync message) 表明这是一个同步报文
versionPTP 2 IEEE 1588-2008 版本
messageLength 44 基础 Sync 包通常是 44 字节
domainNumber 0 默认频道
flagField 0x0200 关键点！ 这里的 Two Step 位被置为 1，说明后面还有一个 Follow_Up 包
correctionField 0 (nanoseconds) 还没经过交换机，所以是 0
sourcePortIdentity 00:0a:35:ff:fe:12:34:56 主钟的唯一身份标识（Clock ID）
sequenceId 123 这个包的序列号
originTimestamp 0.000000000 两步法特殊处： 在 Sync 包里这里填 0，因为精确时间还没“追”上来

2. 紧随其后的：Follow_Up 报文 (UDP 端口 320)

就在上面的包发出去几毫秒后，你会看到另一个包：

  • messageType: 0x8 (Follow_Up message)
  • sequenceId: 123 (必须和上面的 Sync 包一致，否则对不上)
  • preciseOriginTimestamp: 1713863105.123456789
    • 这就是真正的 t1t_1t1​！ 里面精确记录了刚才那个 Sync 包离开网卡的瞬间（17亿多秒，后面跟着纳秒）。

3. 如何在 Wireshark 里一眼看出问题？

作为小白进阶，抓包时重点看这三个“异常信号”：

  1. 看时间间隔：

    如果 Sync 包发得很频繁（比如一秒发 16 个），但 Follow_Up 没跟上，或者 sequenceId 乱了，从钟就会出现“时钟跳变”。

  2. 看 Correction Field：

    如果这个字段一直都是 0，说明你路径上的交换机都不是 TC（透明时钟）。如果你的网络很拥塞，精度一定会变差。

  3. 看 Source Port Identity：

    如果你发现网络里有两个不同的 ID 都在发 Announce 报文，说明出现了两个“老大”，这通常是因为 BMCA 优先级没设好，导致了**“主钟冲突”**。

4. 一个有趣的实战细节：多播地址

你会发现 PTP 的目的 MAC 地址通常是 01:1b:19:xx:xx:xx。

  • 如果是 L2（层 2） 封装，它直接在以太网跑。
  • 如果是 L3（层 3） 封装，它会跑在多播 IP 224.0.1.129 上。
